## The core insight (memorize this, repeat in meetings)

A solo dev built **BrightBean Studio** — a real open-source Buffer/Hootsuite alternative that posts to 10+ social platforms — in 21 days. He did it not by asking AI to "build a product" but by stacking a deliberately boring framework against many well-defined adapter tasks, then running Claude and Codex in parallel against them. The AI never touched architecture. It wrote the 10th Instagram/Bluesky/LinkedIn adapter by reading the previous 9.

## Why you should care this week

If your roadmap still says "v1 in 4 months," a solo dev is out-shipping you at 10x speed on the same scope — not because AI is magic, but because the combination of narrow framework + thin adapters + AI parallelism collapses whole categories of work. If you're a PM on any platform/integrations product (CRMs, payment rails, analytics shims, publishing tools, orchestrators), this is the reference pattern. Rewrite your next scoping doc with it in mind or get undercut.

## The mental model to steal

**"Boring core + many thin adapters"** — the anti-fashion stack is the point.

- **Core**: Django + HTMX + Alpine.js + PostgreSQL + django-background-tasks (no Redis, no React, no Next, no Kafka). Framework handles auth, ORM, templating, job queue — 90% of the product comes free.
- **Adapters**: one Python file per external platform in `providers/` (e.g., `providers/instagram.py`, `providers/bluesky.py`). Every adapter implements the same tiny interface.
- **AI's job**: write adapter N+1 by pattern-matching on adapters 1..N plus the platform's public API docs. That's it.
- **Human's job**: define the interface once, review each adapter's diff, fix auth edge cases.

The reason this works: AI is great at "write thing that looks like these other things" and terrible at "design a system." A boring mainstream framework minimizes the surface area it has to reason about. Fashion stacks (React + Next + Redis + 5 services) multiply that surface. You're trading dev-ergonomics you don't need for an AI-hostile codebase.

## Your 90-minute experiment

Don't build anything grand. Feel the pattern.

1. **Start a fresh Django project** (`django-admin startproject demo`, ~15 min)
2. **Create a providers/ folder** with one file: `providers/base.py` defining `class Provider` with a single method `def post(self, text: str, media: list) -> dict`
3. **Open Claude Code in the repo** and prompt: *"Read providers/base.py. Write providers/bluesky.py implementing Provider using the Bluesky AT Protocol public API. Include auth via app password from env vars."* (~20 min)
4. **Then prompt for providers/mastodon.py** — same sentence, swap the platform. (~15 min)
5. **Wire a one-view /post form** that invokes whichever adapter. (~20 min)
6. **Actually post something from each.** (~10 min)

Total: 90 min. Cost: $0 (Bluesky and Mastodon are both free-tier). Deliverable: two real posts on real networks from code Claude wrote with you driving the interface. You will feel why the pattern scales to 10 platforms in 21 days — and you'll never scope a roadmap the same way again.

## Three claims you should now be able to defend

1. **"The 10x speed came from parallelism of well-defined work, not AI writing novel logic."** If someone asks "what did AI actually do?", you can answer: it wrote ~10 near-identical adapter files against a human-specified interface, plus CRUD boilerplate. Nothing novel.
2. **"Boring framework choices are a force multiplier with AI."** Counterintuitive to most engineers who chase new stacks. The reason: Django has ~15 years of training data, HTMX avoids a whole SPA layer, django-background-tasks avoids Redis — fewer moving parts = fewer places for AI to hallucinate.
3. **"The human role shifts from implementer to interface-designer + reviewer."** You define Provider, you review the diff, you fix the weird auth corner. You are the architect; AI is the typist with perfect recall of the last 9 adapters.

## What you could steal for your own product

- The `providers/` directory pattern for any multi-integration product (CRM syncs, payment gateways, observability backends)
- django-background-tasks if you're small — kills your Redis ops dependency entirely
- Docker + Caddy + Gunicorn as a deploy stack (~40 lines of config, runs anywhere)
